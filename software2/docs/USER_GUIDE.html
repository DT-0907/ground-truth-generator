Ground-truth generator · YOLOv8 + ByteTrack

Turn raw CCTV footage into corrected, training-ready traffic data.

CCTV-YOLO is a native desktop app that detects and tracks vehicles in your camera footage, lets you fix the model's mistakes by hand, and feeds those corrections back into training a better model — over and over, until the model is good enough to do most of the work for you.

9 purpose-built tabs 10 export formats Human-in-the-loop correction Active-learning training loop Runs offline, on your machine

What problem does it solve?

Traffic researchers sit on enormous archives of CCTV video — thousands of hours of intersections, roads, and lots. That footage is full of useful data (vehicle counts, turning movements, speeds, classes), but it is locked inside pixels. An off-the-shelf object detector like YOLOv8 can start to unlock it, but a generic model trained on web photos makes mistakes on your specific cameras: it misses occluded cars, confuses trucks for buses, loses a vehicle's identity behind a sign post, and counts the same car twice.

CCTV-YOLO closes that gap. You run automatic detection across your footage, then a human reviewer corrects the errors in a fast keyboard-driven editor. Those corrections are the ground truth. You build a dataset from them, fine-tune a model, and the next batch of footage needs far fewer corrections. Each pass produces both a cleaner dataset and a smarter model — the active-learning flywheel.

Who is it for?

Traffic and transportation researchers, transportation engineers, and anyone who needs defensible, accurate vehicle-traffic ground truth from existing camera footage — not approximate estimates, but corrected, auditable per-track data they can export to standard annotation and training formats.

The two ideas that drive the whole app

Human-in-the-loop correction

The model proposes; the human disposes. Detection output (tracks) is never overwritten — your edits live in a separate corrections layer. You can always see what the model did versus what's true, and roll back.

Active learning

You don't have to correct everything. The app ranks sessions by how uncertain the model is, so you spend your time where it matters most. Each correction round feeds a training round, and the model gets better at the cases you already fixed.

Note

This is a native PySide6/Qt desktop application — not a website or a Flask server. It runs entirely on your computer, works offline, and stores everything in a folder you control.

Installation & first launch

CCTV-YOLO ships as a self-contained desktop app for macOS and Windows. The bundle includes Python, Qt, OpenCV, PyTorch, and YOLOv8 — there is nothing to install separately.

macOS

  1. Open the CCTV-YOLO.dmg you were given.
  2. Drag CCTV-YOLO.app into your Applications folder.
  3. Launch it from Applications (or Spotlight). On first launch macOS Gatekeeper may ask you to confirm — right-click the app and choose Open if so.
Note · First launch on another Mac

The .app is ad-hoc signed, not notarized. On a Mac other than the one it was built on, the first launch may be blocked by Gatekeeper — right-click the app → Open once to allow it. If macOS instead says the app is "damaged and can't be opened", that's the same Gatekeeper block: right-click → Open clears it (you only need to do this once).

Note · Apple Silicon

On M-series Macs, GPU acceleration through Apple Metal (MPS) turns on automatically — no driver, no setup. The build is a native arm64 bundle, so detection runs roughly 10× faster than CPU.

Windows

Two distribution shapes exist, depending on what you were handed:

  • Single-file installer — run CCTV-YOLO-Setup.exe and follow the prompts. It installs the app and creates a Start-menu / desktop shortcut. This is the easiest path.
  • Folder build (ZIP) — extract the ZIP anywhere, then double-click CCTV-YOLO.exe inside it.
Building from source · Python 3.10–3.12 required

If you are building the Windows app (not just running a prebuilt one), you need Python 3.10, 3.11, or 3.12. Python 3.13 / 3.14 have no PyTorch wheels, so a build on them will hang. The build script now refuses unsupported Python versions, prefers the py -3.12 launcher, and will automatically recreate a build venv that was made with an unsupported Python. And if no supported Python is found at all, the script offers to auto-install Python 3.12 for you — downloaded from python.org and installed per-user (no admin rights), after which the build continues with it automatically. So even on Python 3.14 you don't have to do anything manual: just confirm the prompt (it defaults to Yes).

If you used the ZIP: run Unblock.bat first

When you download and extract a ZIP from a browser, Windows attaches a Mark-of-the-Web (MOTW) flag to every extracted file. On first launch, SmartScreen and Defender will run a full scan of the ~2.5 GB PyTorch/CUDA bundle, which can stall startup for 30–60 seconds.

  • Double-click Unblock.bat once after extracting. It strips the MOTW flag from every file (safe and reversible — it changes no file content), so the app launches as fast as any local program from then on.
If the app won't open at all

Run CCTV-YOLO-debug.bat. It launches the console-enabled build, keeps the window open so you can read native crash messages (DLL load failures, Qt platform-plugin errors, OpenMP conflicts) that never reach the normal crash handler, and writes the full output to startup-output.log next to the exe. Send that log if you need help.

The first-run wizard

The very first time you launch — detected by the absence of a .first_run_complete marker in the data folder — a short wizard walks you through:

  1. Welcome — what the app does.
  2. Data folder — choose where your videos, corrections, models, and exports live (see the next section).
  3. Model download — optionally fetch the small yolov8n base model so you can start detecting immediately.
  4. Tour — a quick overview of the tabs.
  5. Finish — writes the marker so the wizard doesn't show again.

GPU / CUDA at a glance

  • NVIDIA GPUs are auto-detected. The Windows build probes your card with nvidia-smi and picks the right CUDA wheel automatically, so detection uses the GPU when a compatible driver is present (~20× faster than CPU).
  • Apple Silicon uses MPS automatically.
  • Newer GPU generations need a newer CUDA runtime variant in the build, and a recent NVIDIA driver — for example, RTX 50-series (Blackwell) GPUs require cu128 (CUDA 12.8+). The full story — including how to tell why the app fell back to CPU — is in GPU / CUDA & performance.

Requirements

  • macOS 12+ or Windows 10/11.
  • Optional: an NVIDIA GPU with current drivers for ~20× faster inference; Apple Silicon for ~10×.
  • Optional: Tailscale + an SMB share for NAS-mode video access (configured in File → Settings).
  • (For building from source only: Python 3.10–3.12 — 3.13/3.14 have no PyTorch wheels and are rejected by the build script.)

The data folder

Everything you create lives in a single portable folder named cctv-yolo/, completely separate from where the .exe / .app itself is installed. You can move that folder (Documents → Desktop → an external drive) and the app finds it again on the next launch.

Layout

├── data/
│   ├── videos/         raw input videos (.mp4 .mov .avi .mkv)
│   ├── tracks/         YOLO + ByteTrack detection output JSON
│   ├── corrections/    your edits (saved separately — never
│   │                   overwrites tracks/); .bak/ keeps 5 versions
│   ├── exports/        COCO / YOLO / CVAT / MOT / CSV / MP4 / PDF / zip
│   │                   + insights/ + model_compare/ subfolders
│   └── training/       dataset builds (ds_*) + run logs
├── config/             model_config.json, nas.json, ui_state.json,
│                       session_groups.json, training_history.json,
│                       batch_registry.json, batch_session_map.json,
│                       visual_index.json/.npy
├── models/             *.pt + sidecar .meta.json for trained models
├── logs/               app.log (rotating, 5 MB × 5 files)
├── live/               snapshots/ + recordings/ from the Live tab
├── .first_run_complete marker (presence = wizard already shown)
└── crash.log           written only if startup crashes

How the path is resolved & remembered

On every launch the app picks the data root in this priority order:

  1. $CCTV_YOLO_DATA_DIR environment variable — an explicit override that wins over everything else.
  2. The path remembered from last launch. This marker is stored in the OS app-config location — not inside the data folder — so it survives even if you move or delete the data folder:
    • macOS: ~/Library/Application Support/CCTV-YOLO/data_root.txt
    • Windows: %APPDATA%\CCTV-YOLO\data_root.txt
    • Linux: ~/.config/CCTV-YOLO/data_root.txt
  3. Source mode only (running python run.py from a checkout) — the repo root.
  4. Auto-detect — the first existing folder named cctv-yolo in ~/Documents, ~/Desktop, or your home directory.
  5. Default — create ~/Documents/cctv-yolo/.
Tip · moving the data folder

Moved the folder and the app can't find it? Either set CCTV_YOLO_DATA_DIR before launching, or delete the remembered data_root.txt marker — the next launch re-searches and saves whatever it finds.

"Open Folder" buttons everywhere

Every tab has a small OpenLocationBar in its top-right corner. Each button reveals the relevant folder (or highlights a specific file) in Finder/Explorer with one click — so you are never hunting for "where did that export go?". Tooltips show the absolute path. The same shortcuts live in File → Settings → Local Folders, and Help → Show Log Folder jumps straight to logs/.

Preprocessing Tab 1

The Preprocessing tab is where you run automatic detection + tracking on a single video and watch it happen. It combines a video grid, a YOLO model picker, and live processing controls. Use it to spot-check a clip, dial in a confidence threshold, or process one important video carefully before committing to a bulk run.

Walkthrough

  • Model picker — a dropdown of every .pt in your models/ folder, plus a Browse… button to pick one from elsewhere. This is how you choose which model does the detecting — your latest fine-tuned model or a stock yolov8n/s/m.
  • Confidence slider — the minimum detection confidence. Lower it to catch more (and more uncertain) detections you'll review; raise it to keep only high-confidence boxes.
  • Video grid — every video in data/videos/ as a card. Each card shows a status badge (Queued / Processing / Done / Error) using the app's status colors, and a per-video Process control with a live progress bar and ETA you can cancel mid-run.
  • ROI drawing — draw regions of interest on the selected video's preview before processing, so counts and analytics are scoped to the lanes/approaches you care about (see ROIs).
  • Recent runs — a sidebar of what you just processed, so you can jump straight into reviewing it.
  • When a video finishes, the card offers Review, which opens the correction editor for that session.
What it produces

Processing writes a tracks JSON to data/tracks/: per-frame bounding boxes in [x1, y1, x2, y2] (YOLO xyxy) format, grouped into tracks by ByteTrack. This is the raw model output — your starting material for correction.

Ground-truth quality

For ground truth you generally want to detect more than you'll keep — set the confidence a little low so borderline vehicles surface as boxes you can confirm or delete, rather than silently missing them. A missed vehicle is invisible during review; an extra box is one keystroke to remove.

Batch Tab 2

The Batch tab is the workhorse for processing many videos at once. Point it at any folder on disk and it builds a lazy tree view of every video inside, then chews through them in parallel with a scheduler you can resize on the fly. This is how you turn thousands of unread clips into tracks JSON without babysitting each one.

Walkthrough

  • 📁 Pick Folder… — choose a source folder. The app builds a QTreeView of every video in it (and subfolders) and auto-ingests them so you can start immediately. ⟳ Rescan re-reads the tree after you add files.
  • Parallel workers — a spinbox from 1 to 100. This sizes the thread pool: how many videos process simultaneously. Raise it on a strong GPU/CPU; lower it if the machine gets sluggish. You can change it mid-run.
  • Stop after — process only the first N videos, then stop cleanly. Perfect for processing a "first chunk" of a huge archive (e.g. the first 100) before committing the whole batch.
  • Confidence — the detection threshold for the whole batch.
  • ▶ Start All / ⏸ Pause / ✕ Cancel / 🧹 Clear Done — run, pause, atomically cancel in-flight work, and clear finished rows. Cancel is clean: partially-processed videos don't leave corrupt output.
Persistence

Your active folder and which tree nodes were expanded are remembered in config/batch_registry.json. Every batch session is also recorded in config/batch_session_map.json so the app can always find the underlying video file regardless of where it lives on disk — including on a NAS share.

Ground-truth quality

Bulk detection is cheap; reviewing is expensive. Process broadly here, then let the Training tab's uncertainty ranking and the Correction tab's filters point you at the sessions actually worth a human's time.

Correction & Review Tab 3

This is the heart of the app — where a human turns raw model output into ground truth. The Correction tab itself is a dispatcher: it lists every session with stat cards (Total Sessions, Needs Review, Corrected) and filter buttons (All / Needs Review / Corrected). Click Review on a session — or Next Review to walk the queue — to open the full-screen Review window editor.

The Review window editor

A separate window with a toolbar on top, the video canvas in the center, a track sidebar on the right (resizable splitter), and frame controls along the bottom (play, step, slider, timeline minimap). Everything is keyboard-driven — see shortcuts.

  • Modes — V Select, B Draw box. Draw a new box to add a missing detection.
  • Track edits — D Delete a bad track, C change its Class, M Merge two tracks that are the same vehicle, S Split a track that wrongly fused two vehicles.
  • Box propagation — N / P copy the current box to the next / previous frame, so fixing a drifting box takes seconds.
  • Gap interpolation on merge — when you merge two fragments separated by frames, the bbox is linearly interpolated across the gap and drawn dashed, so an occluded vehicle becomes one continuous track.
  • ROIs — Shift+R rectangle, Shift+P polygon (double-click to close).
  • Undo / Redo — Ctrl+Z / Ctrl+Y (snapshots of {tracks, rois}).
  • Save — Ctrl+S.

Non-destructive saves

Why this matters

Your edits are written to data/corrections/ as a separate layer with a "_version": 2 schema — the original tracks/ file is never overwritten. Saves are atomic (a power loss mid-write can't corrupt the file) and keep the last 5 rolling .bak backups. You can always compare what the model produced against what you corrected (that's exactly what the Before/After renderer uses), and you can recover an earlier version.

Exporting from the Review window

The Export… button opens a grouped menu with ten formats (in-memory state is used, so unsaved edits are included). Each export lands under data/exports/<session>/<format>/ and the folder is revealed when done. Full descriptions of each format are in Export formats.

Performance Tab 4

Performance answers "what's actually in this footage, and how good is my detection?" It shows traffic counts and detection statistics, and bundles three heavier analyses. It has a Single / Group toggle so you can look at one session or roll up a whole group (see Groups).

Walkthrough

  • Session / Group dropdown with a provenance badge — it tells you whether the numbers come from your Corrections (and how many edits) or fall back to raw tracks. Never guess which data you're looking at.
  • Stat cards + breakdown tables — total counts and per-class counts, color-coded by vehicle type (car, truck, bus, motorcycle, bicycle…).
  • ROI panel (single mode) — draw/select ROIs so counts are scoped to specific lanes or approaches.
  • Model A/B Compare — run two models on the same video and diff their counts, mean confidence, and per-class breakdown side-by-side. Use it to prove a new model is better before you trust it.
  • Before/After — render raw tracks vs. your corrections side-by-side with a divider; frames whose box sets differ get a red corner marker so the diff is obvious while scrubbing.
  • Confusion Matrix — per-class precision/recall/F1 against your corrections as ground truth.
  • Group membership panel (group mode) — see/manage which sessions are in the group.
Ground-truth quality

A confusion matrix here is your fastest read on dataset trouble: a hot off-diagonal "truck → bus" cell means your model (and possibly your corrections) are inconsistent about that distinction — fix it before you train, or the model will learn the confusion.

Analytics Tab 5

Analytics turns a corrected session into research outputs. Its defining principle: every result renders inline — no "saved to disk and now invisible" outputs, no orphaned status labels. Before its first run each section shows a hint ("Click the button above to compute…"). It runs in Single, Group (aggregate over a group), or Multi-select mode.

The nine sections

Path-density heatmap

A PNG showing where vehicles travel most — instantly reveals the dominant paths through the scene.

OD matrix

Origin–destination counts between ROIs: how many vehicles went from approach A to exit B. Written as CSV.

Time-series CSV

Per-bucket counts (e.g. per-minute) — the raw material for flow charts and peak-hour analysis.

Speed estimation

Per-track speeds, using a pixels-per-meter value. Calibrate it automatically (see Calibration).

Direction of travel

Movement direction analysis for the tracks in the scene.

Annotated MP4

Burns boxes, track IDs, ROI overlays, and a live counter into a video — optionally blurring license-plate regions.

Event clips + supercut

Extracts a short clip per event (ROI entry/exit, anomaly, long dwell, speed outlier, user-flagged) and stitches a highlight reel.

Before/after playback

The same raw-vs-corrected diff renderer as Performance, for visual QA of your edits.

HTML session report

A self-contained report: header, summary stats, embedded heatmap, OD table, time-series, top-10 speeds, anomalies — no template engine, opens in any browser.

Where outputs land

Heatmaps are PNG, OD/time-series are CSV, reports are HTML, clips and annotated video are MP4 — all under data/exports/. The OpenLocationBar reveals them.

Insights Tab 6

Insights is your "is this data good enough to train on?" dashboard. It has four sub-tabs that apply the same three panels — Dataset Health, Anomaly Detection, Confusion Matrix — at different scopes:

Sub-tabScope
SessionOne chosen processed session.
GroupEvery session inside a named group.
DatasetA built YOLO dataset folder under data/training/.
MultiAn ad-hoc multi-select of sessions for one-off comparisons.

The three panels

  • Dataset Health — class & subclass balance, bbox-size distribution (small/medium/large by COCO conventions), median area & aspect ratio, frame coverage, and train/val split counts. It raises plain-English warnings: "Class X is 1.5%", "85% imbalanced", "<200 total bboxes", "small bboxes dominate", "no val set yet". Run this before every training run to catch bias.
  • Anomaly Detection — builds a per-ROI, per-hour-of-day baseline (count, class-mix, mean speed) across all sessions and flags any session whose values fall outside mean ± N·σ. Time-of-day is inferred from the session's processed-at timestamp plus the event's offset in the clip. Without ROIs it falls back to whole-frame stats.
  • Confusion Matrix — held-out evaluation: pick a session whose corrections are kept aside, run the model fresh on the video, match predictions to corrections per-frame by IoU (≥ 0.5 = true positive; class mismatch = confusion; unmatched corrections = false negatives; unmatched predictions = false positives), and render the matrix and per-class metrics.
History

Each panel's results are saved under data/exports/insights/… and re-loadable from a History dropdown beside the panel.

Training Tab 7

The Training tab is where corrections become a model. It runs the active-learning loop: rank what to review, build a dataset from corrections, fine-tune a YOLO model, and promote the result.

Active-learning queue

The top table ranks uncorrected sessions by uncertainty — sessions with low mean confidence, lots of short fragments, and many needs_review flags score highest (most interesting to review). Sorted worst-first, it tells you exactly where your next correction effort buys the most accuracy.

Build a dataset

Choose a build mode:

ModeWhat it includes
All correctedEvery session that has corrections.
Unused since last buildOnly sessions corrected since your last successful training build — the active-learning incremental dataset.
Manual selectionAn explicit list of sessions you pick.
Combine existing datasetsMerge previously-built ds_* datasets into one.

Build parameters: Sample every N frames (annotation stride), val split (0.05–0.30, default 0.10), Filter to ROI (only annotate tracks intersecting an ROI), Base model (yolov8n/s/m/l), Epochs (default 30), and Image size. The build samples annotated frames into a YOLO-format dataset (images/ + labels/ + data.yaml with your class names) under data/training/ds_<id>/, with an automatic train/val split.

Train & promote

Training runs in-process via the Ultralytics API and streams loss and metrics into the UI so you can watch progress live. (This works inside the packaged app — the older subprocess approach that shelled out to the yolo CLI only worked in a dev checkout.) When it finishes, the best weights (best.pt) are copied into models/ as a timestamped .pt, and you get a promote prompt to make it the active model. The model pickers in Preprocessing, Live, and Models pick it up automatically.

Why "Unused since last build" is the key

It means each training round only re-learns from the new work you've done — no re-feeding the same sessions, no manual bookkeeping. The app reads training_history.json to know what's already been used. This is what makes the flywheel in the complete workflow turn cleanly.

Models Tab 8

Models manages the .pt files in your models/ folder and runs head-to-head comparisons.

Walkthrough

  • Models table — columns for filename, size, inference speed (ms/frame, cached), an Active badge marking the current default, and source provenance (from a .meta.json sidecar) so you can see which dataset a trained model came from.
  • Right-click a model to Delete or Rename (rename keeps the active flag in sync).
  • Download YOLOv8… — fetches stock weights from Ultralytics. The downloader is a hardened pure-Python implementation: retries with backoff, two mirrors (GitHub Releases, then HuggingFace), resume support for interrupted downloads, and plain-English errors instead of opaque curl codes.
  • A/B comparison — run two models against either a single video or a dataset's val split, and see the diff. Every comparison is saved as JSON under data/exports/model_compare/ and re-loadable from a History picker.
Ground-truth quality

Comparing on a val split (held-out frames the model never trained on) is the honest way to know whether iteration N+1 is genuinely better than iteration N — not just better at memorizing the training frames.

Live Tab 9

Live runs detection on a real-time stream — a webcam or an RTSP camera URL — for monitoring and quick model sanity checks. It's deliberately minimal and research-grade.

Walkthrough

  • Source — pick a webcam index or paste an RTSP URL. The last source (and its ROIs) are remembered per source.
  • A/B model compare on live — run two models on the same frames side-by-side; model B can be throttled to every Nth frame to save compute.
  • ROIs + alerts — draw ROIs on the live view and get entry / exit / dwell alerts with per-ROI live counts.
  • Recording — record manually, or automatically on an event (e.g. when a vehicle enters an ROI). Recordings and snapshots land in live/.
  • Overlays & resilience — an FPS / inference-ms overlay, a 10-second RTSP connect-timeout toast, and an auto-reconnect banner with exponential backoff for flaky streams.
Note

Live is for monitoring and spot-checking a model in the field. The ground-truth pipeline itself runs on recorded files through Preprocessing/Batch → Correction → Training.

Cross-cutting features

These features aren't a single tab — they thread through the whole app.

Groups

Tag sessions by an attribute — Snow, Night, Intersection-3, whatever you research by — and the app rolls aggregated stats up through Performance, Analytics, Insights, and Training. Create, rename, recolor, and add/remove members; groups are stored in config/session_groups.json. Use them to compare conditions ("are night counts anomalous?") or to build a dataset from only one condition.

ROIs — draw once, respected everywhere

Regions of interest (rectangles or polygons) are drawn in Preprocessing or Correction and then respected by every downstream tab and every export: per-ROI counts, OD matrices, ROI-scoped dataset builds, live alerts. Multiple ROIs on a canvas get distinct palette colors. ROIs are why a "count" can mean "northbound left-turn lane" instead of "the whole frame" — essential for traffic ground truth.

Occlusion / track-gap handling

When a vehicle passes behind a sign, pillar, or another vehicle, the tracker often splits it into two fragments. The app finds likely "same vehicle through a gap" pairs and presents a ranked merge-candidate dialog with a confidence score and predicted gap location. Accept a pair and it routes through the normal merge code — gap frames are interpolated and tagged "occluded": true so they show up distinctly in the timeline minimap and on the canvas. This keeps one physical vehicle as one track, which is what counts and trajectories depend on.

Cross-session visual / CLIP search Ctrl+F

Two search surfaces:

  • Attribute search — find tracks across all sessions by class, ROI membership, time-of-day, confidence range, or track length (a pure-data scan, no video reads; results link back to a review window).
  • Visual similarity — build an embedding index over vehicle crops, then query by image (pick a track, find similar ones everywhere) or by text ("red truck", "police car") when CLIP is available. It tries open_clip first (best, text + image), falls back to a ResNet50 backbone (image-only), and surfaces a helpful message if neither is installed. Cached at config/visual_index.json/.npy.

Anomaly detection

Learns a baseline of normal traffic per ROI and hour-of-day across your sessions, then flags outliers. See the Insights tab.

Auto-clips & supercut

From the Analytics tab, extract a short annotated MP4 per event and optionally concatenate them into one highlight reel. Event types: roi_entry, roi_exit, anomaly, long_dwell, speed_outlier, user_flag.

Calibration (pixels-per-meter)

Speed estimation needs a pixels-per-meter scale. The app computes it two ways and reports both so you can choose: a fast track-velocity heuristic (assumes median car speed ≈ 13.4 m/s / 30 mph), and a more accurate vanishing-point geometry method (Hough lines → RANSAC vanishing point → median car bbox height ≈ 1.5 m sedan).

Alerts

A simple rule engine (used by the Live tab) that tracks per-track state and fires entry / exit / dwell alerts.

GPU / CUDA & performance

Detection is the slow part. A GPU makes it roughly 20× faster (NVIDIA/CUDA) or 10× faster (Apple MPS) than CPU, so getting the GPU working is the single biggest speed lever.

How to tell what device is in use

  • The status bar shows the active device (e.g. NVIDIA GeForce RTX 3060 (CUDA 11.8), Apple Metal (MPS), or CPU).
  • Help → About shows the same device string alongside version and platform.
  • Processing logs record it too, in one consistent voice.

The three reasons it falls back to CPU

If you see "CPU" when you expected the GPU, the app tells you which of these you hit:

  1. CPU-only PyTorch was bundled. The build doesn't contain CUDA support at all → rebuild with a CUDA variant (the Windows build script defaults to one).
  2. CUDA PyTorch is bundled, but no usable NVIDIA GPU is visible. Usually the NVIDIA driver is older than the bundled CUDA runtime needs → update your driver, or rebuild with a CUDA variant matching your driver.
  3. Driver too old for the bundled CUDA runtime. The most common Windows case — nvidia-smi shows the GPU, but the app runs on CPU because the wheel's CUDA runtime needs a newer driver.

A related case: your GPU is newer than the CUDA build that was bundled (e.g. an RTX 50-series Blackwell card running a cu124 build). The app now detects this and reports CPU-with-a-reason — telling you to "rebuild with cu128" — instead of crashing.

NVIDIA: matching the CUDA variant to your driver

The Windows build can be produced with different CUDA wheel variants. The build script auto-detects your GPU via nvidia-smi and picks the right wheel automatically; you only need to think about variants when building for a different machine or overriding the choice. Newer variants support newer GPUs but require newer drivers:

VariantDriver / GPU notes
cu118Broadest compatibility — works with any NVIDIA driver ≥ 452.39 on Windows. The safe fallback for older cards / mixed hardware.
cu121Newer GPUs; needs driver ≥ 528.33.
cu124Newer still; needs a more recent driver.
cu126Recent cards; needs a recent driver.
cu128Required for RTX 50-series (Blackwell) — CUDA 12.8+; needs the newest driver.
cpuNo GPU; ~250 MB instead of ~2.5 GB.
Newest GPUs (Blackwell / RTX 50-series)

RTX 50-series (Blackwell) cards require cu128 (CUDA 12.8+) and the newest NVIDIA driver. Older cards use cu124, cu121, or cu118. The build auto-detects the right one; if a card is newer than the bundled build, the app reports "running on CPU — rebuild with cu128" instead of crashing.

Tip · overriding the auto-detected variant

To force a specific wheel (e.g. building on a machine without the target GPU, or distributing to mixed hardware), set the CCTV_YOLO_TORCH_VARIANT environment variable before building — e.g. cu128 for Blackwell, or cu118 for the widest driver compatibility.

Apple Silicon (MPS)

On M1/M2/M3 Macs, Metal acceleration is enabled automatically via torch.backends.mps — no install, no configuration. The status bar will say Apple Metal (MPS).

Troubleshooting

The exe won't open (Windows)

Run CCTV-YOLO-debug.bat. It launches the console-enabled build, keeps the window open, and captures every line of output — including native errors (DLL load failed, OpenMP conflicts, Qt platform-plugin failures) that never reach Python's crash handler — into startup-output.log next to the exe. Read that log to see exactly what failed.

The Windows build hangs or won't install PyTorch

You're almost certainly building with an unsupported Python. PyTorch publishes no wheels for Python 3.13/3.14, so a build on them stalls. Use Python 3.10, 3.11, or 3.12 — the build script now refuses newer versions, prefers the py -3.12 launcher, and auto-recreates any build venv that was made with an unsupported Python. If no supported Python is installed, the script offers to auto-install Python 3.12 (downloaded from python.org, installed per-user with no admin rights) and then continues automatically — so on Python 3.14 you just confirm the prompt, which defaults to Yes, and do nothing else by hand.

First launch is very slow

That's the Mark-of-the-Web scan. Run Unblock.bat once after extracting the ZIP and subsequent launches are fast (see Installation).

GPU isn't being used

Check the status bar / About dialog for the device string and the CPU-fallback reason, then follow the matching fix in GPU / CUDA — almost always "update your NVIDIA driver".

Where the logs live

  • app.log — the rotating runtime log (5 MB × 5 files) in <data folder>/logs/. Jump there via Help → Show Log Folder.
  • crash.log — written at the data-folder root only if startup itself crashes.
  • startup-output.log — produced by CCTV-YOLO-debug.bat, next to the exe.

The app can't find my data

You probably moved the data folder. Set CCTV_YOLO_DATA_DIR to its new location before launching, or delete the remembered data_root.txt marker so the app re-searches (see The data folder).

The complete workflow ★ Playbook

From thousands of unread, batched videos to a trained model — and back again. This is the active-learning flywheel, end to end. Each loop produces a cleaner dataset and a smarter model, and the correction effort drops every time around.

The flywheel in one sentence

Detect everything cheaply → correct the most uncertain ~100 sessions by hand → build a dataset from only those new corrections → train → promote the better model → run it on the next batch (which now needs fewer corrections) → repeat.

A
data/videos/ · or Batch tab · or NAS

Ingest — get the footage in reach

Drop your archive of videos into the data/videos/ folder, or simply point the Batch tab at any folder on disk (or a Tailscale-mounted NAS share). You don't need to copy thousands of files in — Batch reads them where they live and remembers the mapping.

Tip: Keep videos on a fast local drive or a well-connected NAS — disk read speed is part of throughput.
B
Batch tab (Ctrl+4)

Bulk auto-detect — let YOLO do the first pass

In Batch, set Parallel workers as high as your hardware allows, optionally set Stop after to process a first chunk (say 100 videos), and hit ▶ Start All. Every video gets a tracks JSON. On the first iteration use a stock model (yolov8n from the wizard download); on later iterations use your promoted model.

Tip: Process the whole archive if you can — detection is cheap and you can review only a subset. More tracks = more material to mine with search and analytics later.
C
Correction tab (Ctrl+2) → Review window

Review & correct the first ~100 sessions — make ground truth

This is the human-in-the-loop core. In Correction, filter to Needs Review (or let Training's uncertainty ranking tell you what's worst), open the Review window, and fix: tighten boxes, correct classes (C), Merge/Split tracks, accept occlusion merges, and draw ROIs for your lanes/approaches. Save with Ctrl+S. Saves are non-destructive — the raw tracks survive.

Tip: Be consistent about class boundaries (van = car? pickup = truck?). Consistency in your corrections is what the model actually learns — check the confusion matrix to catch your own drift.
D
Training tab (Ctrl+6) → "Unused since last build"

Build dataset #1 — turn corrections into a YOLO dataset

In Training choose build mode Unused since last build and click build. It includes only the sessions you just corrected, samples annotated frames (your stride), applies the val split, and writes data/training/ds_001/ with images/, labels/, and data.yaml. Before building, run Insights → Dataset Health to catch class imbalance.

Tip: A bigger sample stride = smaller, faster dataset but fewer examples. Start around "every 5 frames" and adjust based on dataset health.
E
Training tab → Train + promote prompt

Train model #1 — fine-tune and promote

Pick a base model and epochs, then start training. The app trains in-process via the Ultralytics API and streams progress live. When it finishes, the best weights are copied into models/ as a timestamped .pt — accept the promote prompt to make it the active model so the Preprocessing/Batch/Live pickers use it.

Tip: 30 epochs on yolov8n is a fast first loop. Scale up base model and epochs once you have more data and know the pipeline works.
F
Performance (Ctrl+3) · Insights (Ctrl+9) · Models (Ctrl+7)

Evaluate — prove the new model is actually better

Use Insights → Confusion Matrix and Dataset Health to read class-level accuracy and balance. In Models → A/B compare, pit your new model against the stock yolov8n on a dataset val split (held-out frames) so the comparison is honest. Use Performance → Before/After to eyeball the difference on a real clip.

Tip: Always compare on a val split, never on training frames — that's the difference between "memorized" and "learned".
G
Back to Batch with the promoted model → repeat C–F

Repeat — turn the flywheel

Point the better model at the next batch of unread videos (Batch tab, with the promoted model selected). Correct the next ~100 sessions — now faster, because the model is better and fewer boxes need fixing. Run Unused since last build again: it creates ds_002 from only the sessions corrected since the last train. Optionally Combine existing datasets to fold ds_001 + ds_002 together, then train model #2. And so on.

↺ Each iteration: new dataset · smarter model · less correction effort

Why this loop is the whole point

  • Each iteration creates a new, named dataset (ds_001, ds_002, …) — your work is versioned and auditable, never overwritten.
  • "Unused since last build" prevents re-work — you only ever feed new corrections into the next round, and the app tracks this for you.
  • Correction effort drops every loop — a model trained on your cameras makes fewer mistakes on your cameras, so review gets faster and faster.
  • You can stop any time — when the A/B comparison shows the model is good enough for your accuracy target, you have both a trained model and a corrected ground-truth corpus to export in any of the 10 formats.
Printable cheat-line

Ingest → Batch-detect → Correct ~100 (Needs Review) → Build "Unused since last build" (ds_N) → Train + promote → A/B on val split → point new model at next batch → repeat.
